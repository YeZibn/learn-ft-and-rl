# Module 3: SFT Pipeline

## 3.1 模块目标

掌握监督微调的完整工程流程，理解 SFT 不是“扔数据进去跑一下”，而是一条需要持续诊断的数据与训练闭环。

学完本模块，你应该能回答：

1. SFT 数据长什么样
2. 为什么要清洗、去重、切分
3. 为什么要做数据混合
4. LoRA / QLoRA 在 pipeline 里扮演什么角色
5. 训练时要看哪些指标
6. 怎么判断模型是变强了还是过拟合了

## 3.2 SFT pipeline 的整体流程

一条标准的 SFT 流程通常是：

1. 定义任务
2. 收集或构造数据
3. 清洗和格式统一
4. 切分训练集与验证集
5. 选择模型和模板
6. 配置训练参数
7. 开始训练
8. 监控 loss 和验证集
9. 做样本级评测
10. 进行误差分析
11. 迭代数据和参数

核心理解是：

> SFT 不是模型问题单独完成，而是数据工程、模板工程和训练工程共同完成。

## 3.3 SFT 数据是什么

SFT 数据本质上是：

> 输入 + 标准输出

但在 LLM 场景里，通常不是单条，而是多轮消息。

### 常见形式 1：单轮指令格式

```json
{
  "instruction": "请总结下面这段话",
  "input": "......",
  "output": "......"
}
```

### 常见形式 2：多轮 messages 格式

```json
[
  {"role": "system", "content": "你是一个助手"},
  {"role": "user", "content": "请总结下面这段话"},
  {"role": "assistant", "content": "......"}
]
```

### 常见形式 3：带工具轨迹格式

```json
[
  {"role": "user", "content": "帮我查一下明天上海天气"},
  {"role": "assistant", "content": "我来查询一下。"},
  {"role": "tool", "content": "{\"weather\": \"...\"}"},
  {"role": "assistant", "content": "明天上海......"}
]
```

## 3.4 为什么数据格式统一很重要

如果数据格式不统一，模型学到的行为会乱。

常见问题包括：

1. system prompt 缺失
2. 角色顺序混乱
3. tool message 格式不一致
4. 输出风格漂移
5. 同一任务在不同样本里模板不统一

这些问题会直接影响：

- 收敛速度
- 输出稳定性
- 结构化能力
- 任务泛化

## 3.5 为什么要清洗数据

原始数据通常有很多问题：

1. 重复样本
2. 错误答案
3. 噪声文本
4. 格式不一致
5. 无效对话
6. 低质量长文本
7. 任务目标不明确

如果不清洗，模型会学到这些坏习惯。

### 清洗的目标

不是“把数据变少”，而是：

- 去掉污染
- 保留有效行为
- 提高样本密度
- 提高监督信号质量

## 3.6 去重为什么重要

重复数据会让模型过拟合某些模式。

尤其是：

1. 相同问答重复很多次
2. 某些模板被过度强调
3. 某些回答风格被强行放大

这会导致：

- 模型偏向特定模式
- 泛化变差
- 风格不稳定
- 训练看起来更顺，但实际更脆

## 3.7 为什么要切分 train / eval

如果没有验证集，你不知道模型是真变强了，还是只是记住了训练集。

验证集的作用是：

1. 检查泛化
2. 监控过拟合
3. 比较不同配置
4. 做回归测试

### 关键点

验证集要尽量和训练集分布相似，但不能重复。

## 3.8 为什么要做数据混合

现实任务往往不是单一能力。

比如一个 Agent 可能同时需要：

- 对话
- 代码
- 工具调用
- 数学
- 格式输出
- 反思修正

所以训练集不能只放一种样本。

### 常见混合方式

1. 通用对话
2. 指令跟随
3. 工具调用
4. 推理样本
5. 代码样本
6. 领域任务样本
7. 安全和拒答样本

## 3.9 数据配比为什么重要

数据配比决定模型偏向什么行为。

例如：

- 工具调用样本太少，模型就学不会用工具
- 对话样本太多，模型可能更会聊天，不会做事
- 推理样本太多，可能输出变长但未必更实用

所以数据混合不是简单拼接，而是行为权重设计。

## 3.10 什么叫 full fine-tuning

全参微调就是模型参数整体更新。

### 优点

1. 表达能力强
2. 适合深度调整模型行为

### 缺点

1. 显存开销大
2. 训练成本高
3. 容易把底座能力改坏
4. 不适合快速反复试验

## 3.11 什么叫 LoRA

LoRA 是参数高效微调的一种方式。

它不是直接改所有参数，而是在某些线性层上加低秩适配器。

### 好处

1. 显存更省
2. 训练更快
3. 适合反复试验
4. 很适合 SFT 阶段

### 适用场景

- 大多数入门和实战项目
- 数据量不大时
- 需要快速迭代时

## 3.12 什么叫 QLoRA

QLoRA 是在量化底座上做 LoRA 微调。

### 作用

1. 进一步省显存
2. 更适合单卡或资源有限环境
3. 让大模型微调更可行

### 注意

QLoRA 是为了降低成本，不是为了自动提升能力。  
它解决的是“能不能训”的问题，不是“自动更聪明”的问题。

## 3.13 训练参数为什么重要

SFT 里常见的参数包括：

1. learning rate
2. batch size
3. gradient accumulation
4. warmup
5. max sequence length
6. epochs
7. weight decay
8. scheduler
9. eval frequency

这些参数会直接影响：

- 收敛速度
- 稳定性
- 是否过拟合
- 是否学到格式
- 是否遗忘底座能力

## 3.14 一个最容易犯的错：只看 loss

loss 下降，不代表你想要的行为真的出现了。

你还要看：

1. 验证集表现
2. 样本级输出
3. 格式遵循率
4. 工具调用准确率
5. 长尾样本表现
6. 错误类型分布

## 3.15 SFT 训练中常见的坏信号

1. 训练 loss 很快下降，但验证集不动
2. 模型开始胡乱套模板
3. 工具调用变多但更乱
4. 输出更长但更空
5. 某类任务明显退化

通常意味着：

- 数据有问题
- 配比有问题
- 学习率有问题
- 模板有问题
- 训练过头了

## 3.16 评测不是最后一步

SFT 里最常见的错误是训练完才开始看效果。  
正确做法是：

1. 数据阶段就抽样检查
2. 训练前先设基线
3. 训练中持续看验证
4. 训练后做样本级分析
5. 必要时回改数据

## 3.17 SFT pipeline 的本质

SFT pipeline 的本质不是“让 loss 最小”，而是：

> 用高质量监督样本，稳定地把模型推向你想要的行为分布。

所以 pipeline 的核心工作不是只有模型，而是：

- 数据
- 模板
- 配比
- 训练稳定性
- 评测闭环

## 3.19 核心结论

1. SFT 不是模型问题单独完成，而是数据工程、模板工程和训练工程共同完成
2. 数据格式统一比你想的更重要
3. 去重、清洗、切分、混合是 SFT 的基本功
4. LoRA / QLoRA 是为了让训练更可控、更经济
5. loss 不是唯一指标，样本级行为才是关键
