# Module 4: Agent SFT Data

## 4.1 模块目标

把普通对话数据升级为 Agent 轨迹数据，理解 Agent 数据和普通 SFT 数据的本质差异。

学完本模块，你应该能回答：

1. Agent 数据和普通 SFT 数据有什么不同
2. 什么是 trajectory
3. 为什么工具调用数据不能只存最终答案
4. planning、reflection、correction 分别怎么做
5. 过程监督和结果监督的区别是什么
6. 为什么 Agent 数据质量比数量更关键

## 4.2 核心认知

普通 SFT 数据通常是：

- 输入
- 标准输出

但 Agent 数据通常是：

- 任务目标
- 中间思考
- 动作选择
- 工具调用
- 工具返回
- 再规划
- 最终输出

也就是说，Agent 数据不是单点答案，而是一个决策链条。

## 4.3 什么是 trajectory

trajectory 就是一次任务执行轨迹。

它一般包含：

1. 用户目标
2. 模型当前判断
3. 选择的动作
4. 工具返回结果
5. 根据结果的再决策
6. 最终完成情况

你可以把它理解成：

> Agent 在环境里“怎么一步步做事”的完整记录。

## 4.4 为什么 Agent 不能只训最终答案

如果你只给模型最终答案，它会学到：

- 结果长什么样

但学不到：

- 为什么要这么做
- 什么时候要调用工具
- 出错后怎么恢复
- 如何拆解任务
- 如何在中间判断是否继续

对 Agent 来说，这些过程信息非常关键。

## 4.5 Agent SFT 数据的基本结构

一个典型的 Agent 轨迹样本可以拆成：

1. `task`：任务目标
2. `context`：上下文信息
3. `plan`：初步规划
4. `action`：具体动作
5. `observation`：工具或环境反馈
6. `revision`：修正策略
7. `final`：最终输出

例如：

```json
[
  {"role": "user", "content": "帮我查一下明天上海天气，并告诉我适合带什么衣服"},
  {"role": "assistant", "content": "我先查询天气，再给你建议。"},
  {"role": "tool", "name": "weather_search", "content": "{\"temp\": 18, \"rain\": true}"},
  {"role": "assistant", "content": "明天上海大约18度，有降雨，建议带外套和雨具。"}
]
```

## 4.6 Tool-use 样本应该包含什么

工具调用样本不能只保留“调用了什么工具”，还要保留：

1. 为什么要调用这个工具
2. 参数是怎么决定的
3. 调用后如何解释结果
4. 调用失败时怎么办

一个完整的 tool-use 样本至少要回答：

- 用哪个工具
- 什么时候用
- 参数怎么填
- 返回后怎么继续

## 4.7 Planning 样本是什么

Planning 样本是教模型先拆任务，再执行。

例如用户问：

“帮我安排一个去杭州两天一夜的出行计划。”

合理行为通常是：

1. 识别约束
2. 拆分步骤
3. 查询必要信息
4. 生成计划
5. 输出结果

Planning 数据要训练：

1. 任务分解
2. 依赖顺序
3. 优先级判断
4. 中间检查点
5. 何时执行、何时停止

## 4.8 Reflection 样本是什么

Reflection 是模型在失败或不确定后，对自己行为的复盘和修正。

典型形式：

- 上一步错了哪里
- 为什么错
- 应该怎么改
- 下一步怎么做

Reflection 很重要，因为真实 Agent 经常不是一次做对，而是要修正。

## 4.9 Correction 样本是什么

Correction 样本是对错误答案或错误轨迹的修正版本。

这类数据能教模型：

1. 哪些行为是错的
2. 错了以后正确写法是什么
3. 如何从错误中恢复

## 4.10 过程监督和结果监督

### 结果监督

只看最后答案对不对。

优点：

- 简单
- 省标注
- 容易评估

缺点：

- 看不到中间过程
- 很难定位错误
- 对长链条任务不友好

### 过程监督

不仅看最终答案，还看中间步骤是不是合理。

优点：

- 更适合 Agent
- 能训练中间行为
- 更容易控制工具使用和规划

缺点：

- 标注成本更高
- 设计更复杂

对 Agent 的判断：

> 过程监督通常比纯结果监督更重要。

## 4.11 为什么 Agent 轨迹数据很难做

因为它不只是“写答案”，而是要写“行为链”。

难点包括：

1. 任务目标经常模糊
2. 中间步骤可能有多种正确路径
3. 工具返回是动态的
4. 失败状态很多
5. 一条轨迹里可能混有多个子任务

所以 Agent 数据通常要结合：

- 人工设计
- 规则生成
- 模型合成
- 环境采集
- 人工筛选

## 4.12 合成数据为什么重要

Agent 数据人工标注太贵，所以常常要靠合成数据扩充。

常见方式：

1. teacher model 蒸馏
2. self-instruct
3. 程序化生成
4. 环境日志回放
5. 失败样本修正

但合成数据有风险：

- 容易模板化
- 容易引入教师偏差
- 容易产生“看起来对、实际上错”的样本

所以合成数据一定要配合筛选和校验。

## 4.13 Agent 数据的质量标准

一条好的 Agent 数据，至少要满足：

1. 任务目标明确
2. 中间步骤合理
3. 工具选择正确
4. 参数传递正确
5. 工具返回后能继续处理
6. 最终输出和过程一致
7. 轨迹可复现、可解释

## 4.14 一个实用的 trajectory schema

你可以把 Agent 样本设计成下面这种逻辑结构：

```json
{
  "task": "...",
  "context": "...",
  "plan": ["...", "..."],
  "steps": [
    {
      "thought": "...",
      "action": "...",
      "observation": "..."
    }
  ],
  "final": "..."
}
```

注意：

- `thought` 不一定要原样暴露
- `action` 要结构化
- `observation` 要尽量真实
- `final` 要和过程一致

## 4.15 什么时候要记录思考，什么时候不要

你不一定要把所有内部思考都显式保存。

更稳妥的做法是：

1. 保留过程结构
2. 根据目标决定是否显式使用长思考
3. 不要把无关的冗长思考当成质量

## 4.16 Agent 数据中的常见错误

1. 只有最终答案，没有中间行为
2. 工具调用记录不完整
3. 回答和工具结果矛盾
4. 轨迹中有跳步
5. 多轮任务中上下文丢失
6. 规划和执行互相不一致
7. 样本风格不统一

这些问题会直接伤害模型学习效果。

## 4.19 核心结论

1. Agent 数据的本质不是答案，而是行为轨迹
2. 工具调用数据必须保留过程信息
3. Planning、Reflection、Correction 是 Agent 数据的关键组成
4. 过程监督通常比纯结果监督更重要
5. Agent 数据质量比数量更关键

你可以自己设计三个样本：

1. 一个简单问答样本
2. 一个工具调用样本
3. 一个“先规划再执行再修正”的样本

然后判断：

- 哪个是普通 SFT
- 哪个是 Agent SFT
- 哪个包含了过程监督信息
